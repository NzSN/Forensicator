S3 Architecture — Three Approaches

How to organize 5 recovery tasks (strings, vtables, lists, arrays, chunks) + clustering

A

Monolithic recover module

Single forensicator-core/src/recover.rs with all 5 detectors as functions.

Pros

  • Fast to write
  • No trait design overhead

Cons

  • ~800+ line file
  • Hard to test individually
  • Can't add detectors later without touching core file
B

Tiered by data source

Two modules: strings/ (reads raw bytes from AddressSpace) and shapes/ (analyzes PointerGraph). Heap chunks split between them.

Pros

  • Clear data dependency
  • Strings simple, shapes complex

Cons

  • Heap chunks straddles both
  • Arbitrary split — vtables could go either way
C

Detector trait + catalog

Each recovery task is a module implementing a StructureDetector trait. Outputs collected into a StructureCatalog. New detectors can be added later.

Pros

  • Extensible — add detectors without touching existing code
  • Each detector testable in isolation
  • Matches S1/S2 modular pattern

Cons

  • Trait design upfront
  • More files (5 detector modules + catalog)
