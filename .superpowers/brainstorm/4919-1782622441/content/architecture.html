Three architectural approaches for S2

All build on S1. Pick the structure that feels right.

A

Monolithic graph module

Single forensicator-core/src/graph.rs — packs pointer classification, graph construction, and query API into one file.

Pros

  • Fast to implement
  • Everything visible in one place

Cons

  • ~600+ line file
  • Hard to test in isolation
  • Would need fission later
B

Extend AddressSpace

Add scan/graph/query methods directly onto the S1 AddressSpace type. Single unified memory model.

Pros

  • Single source of truth
  • No new types to thread through

Cons

  • AddressSpace becomes huge
  • Violates single responsibility
  • Hard to version/swap
C

Tiered pipeline (4 modules)

Separate crates/modules for each concern: pattern definition, scanning, graph construction, and query API.

Pros

  • Each component testable independently
  • S3 imports only what it needs
  • Matches S1 modular style

Cons

  • More files to create
  • More trait/interface design upfront
